What It Does

FastAPI yt-dlp PostgreSQL Spotify API Python 3.11+ Vanilla JS FFmpeg psycopg2

⬇ Single Download

Paste any YouTube URL, pull it as audio or video. Output lands in an organized local folder, ID3-tagged and ready.

⊞ Playlist Batch

Download full YouTube playlists into timestamped, sanitized folders. Concurrent jobs run under a semaphore limit — no resource hammering.

↔ Spotify Mirror

Point it at any Spotify playlist, album, or artist. It fetches metadata, finds each track on YouTube, downloads, converts to MP3, and writes history to the database.

01 — Parse

Accept a Spotify URL or URI and identify whether it's a playlist, album, or artist collection.

02 — Fetch

Pull full track metadata from Spotify — titles, artists, duration — using either OAuth or the sp_dc cookie fallback.

03 — Search

Query YouTube per track to find the best audio match. Falls back to YouTube Music lookup for editorial playlists.

04 — Download

yt-dlp pulls the audio stream. FFmpeg converts to MP3. Semaphore limits concurrent jobs.

05 — Tag & Persist

mutagen writes ID3 tags to each file. The job, history, and file path are recorded to PostgreSQL.

Frontend

Static HTML / CSS / JS — no build step, no npm, no runtime framework. Served directly by FastAPI.

API Layer

FastAPI routes split across downloads, Spotify, files, and stats. Clean separation of concerns at the HTTP boundary.

Service Layer

Download orchestration, Spotify logic, and DB access live here as independent service modules.

Persistence

PostgreSQL via psycopg2. Three tables: downloaded_songs, spotify_downloads, jobs. Schema auto-created on startup.

Why no npm?

The frontend is intentionally zero-toolchain. Static HTML, CSS, and JS served directly by FastAPI. No bundler, no transpiler, no node_modules. It's a deliberate constraint that keeps the project deployable and understandable without a separate frontend environment.

Why two Spotify auth strategies?

The Spotify API restricts access to editorial playlists behind OAuth. The sp_dc cookie flow was added as a pragmatic fallback for cases where OAuth isn't the right fit. Supporting both means the tool works in more real-world situations.

Why semaphore-based concurrency?

Unbounded parallel downloads will saturate disk I/O and network quickly. A configurable semaphore cap keeps the system stable under large playlist jobs without needing a full job queue.

Why PostgreSQL for a local tool?

Download history, job state, and mirror records need to survive restarts and be queryable. A flat file would have worked for simple history, but PostgreSQL gives proper job tracking, progress upserts, and storage stats with almost no overhead for a local setup.
